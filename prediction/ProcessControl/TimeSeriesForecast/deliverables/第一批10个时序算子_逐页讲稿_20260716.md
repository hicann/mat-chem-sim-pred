# 第一批 10 个时序算子合入评审材料与逐页讲稿

本文件是合入评审材料的可审阅文本版本。仓库 Check_Pr 不允许在当前路径提交 PPTX 等二进制文件，因此演示内容以 Markdown 形式入库；本地 PPTX 不纳入 MR。

验证环境：Ascend 910B3、CANN 8.1.RC1、PyTorch 2.5.0、torch_npu 2.5.1。

## 使用原则

这套材料有三种性能口径，汇报时不能混在一起：

1. `eager torch_npu -> custom Ascend C`：同一块 NPU 上，比较框架逐步执行与自定义算子执行。这是 9 个算子的历史主性能口径。
2. `TorchAir fullgraph -> custom Ascend C`：同一块 NPU 上，进一步加入图模式强基线。6 个原模型可直接成图；AutoCorr、S6、TiRex 需要人工做数学等价表达改写后才能成图。
3. `CPU fallback -> custom Ascend C`：仅用于 Koopa。原始 `torch.linalg.lstsq` 不支持 NPU，只能回退 CPU，TorchAir 也不能把这个不支持的路径变成 NPU 完整图。

“E2E”也必须说明边界：本批材料中的 E2E 可能是 3 层 encoder、单个 sLSTM layer、一个 validation loop 或 Koopa 指定 forecast 路径，不等于完整训练和所有数据集上的整模型结论。

## 第 1 页：第一批 10 个时序预测算子

### 这一页要表达什么

本次提交不是新增 10 个完整时序模型，而是从不同模型中提取出 10 段 NPU 执行瓶颈，把它们做成独立、可构建、可调用、可测试的 CANN/Ascend C 算子。

这些瓶颈主要来自三类问题：

- 时间递归：后一步依赖前一步，框架通常需要按时间步反复 launch 小 kernel。
- 周期聚合：Autoformer 要做 lag score、Top-K、Softmax、循环位移和加权求和，框架路径由很多小算子组成。
- NPU 能力缺口：Koopa 的 `torch.linalg.lstsq` 不支持 NPU，只能回退 CPU。

### 建议讲法

> 我们没有改模型数学含义，而是把模型中执行代价高或 NPU 不支持的局部步骤做成 Ascend C kernel。评审对象是算子交付件、接口、正确性和明确边界下的性能，不是声称 10 个完整模型都已经替换完成。

## 第 3 页：瓶颈在执行边界，而不只是 FLOPS

### 三条执行路径

#### 1. eager torch_npu

Python 循环按时间步调用 PyTorch 算子。一次循环里的 `matmul`、`exp`、`sigmoid`、`mul`、`add` 等会形成多个 NPU kernel。即使每个 kernel 算得很快，CPU 仍要不断向 NPU 提交任务。

#### 2. TorchAir 图模式

TorchAir 尝试捕获整个 PyTorch 子图并生成 GE 图，能显著减少 eager 调度成本。但时间循环可能被展开成数千到数万个 FX 节点，首次编译时间也可能达到几十秒到十几分钟。

#### 3. 自定义 Ascend C

把时间循环和状态更新写进一个固定 kernel。框架只调用一次 ACLNN，自定义 kernel 在设备内部推进全部时间步。

### 已完成的图模式结论

- 6 个原模型路径可直接 `fullgraph=True` 成图：UnICORNN、SRU、CfC、coRNN、LTC、SelectiveScan1D。
- 3 个原模型路径受 converter 限制：AutoCorr 缺 `aten.roll`，S6 缺 `aten.softplus`，TiRex 缺 `aten.log_sigmoid_forward`。原模型不能直接成图，必须人工修改模型表达后才可运行 TorchAir。
- Koopa 的 `linalg_lstsq` 不支持 NPU，不属于“图模式再优化”的问题。
- 在 9 个可运行图路径上，custom 相对 TorchAir 仍快 `1.39x-7.39x`。

### 为什么图模式仍没有完全替代 custom

图模式主要减少框架调度，但没有自动识别并生成一个固定大小的时序 scan kernel。以正式 shape 为例，捕获节点数从 3,703 到 24,878；首次编译和运行从 69.70 秒到 883.51 秒。对于 3 个 converter 缺失算子，框架方案还要额外维护一份等价表达改写；custom 直接实现目标子图语义，只需在模型适配层替换为 ACLNN 调用。

## 第 4 页：统一落地模式

### Host tiling 是什么

Host 端读取输入 shape，决定：

- 使用多少个 AI Core；
- 每个 core 负责哪些 batch、channel、head 或 sequence group；
- UB 需要分配多少空间；
- 尾块不足完整 tile 时如何处理。

### GM、UB、AI Core 的关系

- GM 是 NPU 全局设备内存，容量大，kernel 能通过 `GlobalTensor` 地址访问。
- UB 是 AI Core 片上的高速缓冲区，容量小，但适合做向量计算和保存递归状态。
- `DataCopy` 负责在 GM 和 UB 之间搬运连续数据。
- `LocalTensor<float>` 是对 UB 中一段 float 数据的张量视图，`Mul`、`Exp`、`Muls` 等 Ascend C API 通常在这种本地张量上执行。

典型数据流是：

```text
GM 输入 -> DataCopy -> UB/LocalTensor -> 向量计算 -> DataCopy -> GM 输出
```

少量标量可以用 `GlobalTensor.GetValue(index)` 读取，但大量向量计算不应把 GM 当普通本地数组直接运算。

### 为什么时间循环仍然存在

递归公式中的 `state_t` 依赖 `state_(t-1)`，数学上不能把时间顺序删除。优化的是“循环放在哪里”：

- 原路径：Python/框架循环，每一步可能启动多个 kernel。
- custom：单个 kernel 内部循环，状态尽可能在 UB 或 kernel 管理的设备内存中复用。

## 第 5 页：SelectiveScan1D

### 在 Mamba 中的位置

一个简化 Mamba block：

```text
输入 -> input projection -> conv1d + SiLU -> 生成 delta/B/C
     -> SelectiveScan1D -> residual gate -> output projection
```

它不是整个 Mamba，也不是整个 block，但它是负责沿时间传播记忆的核心步骤。

### 输入和输出

```text
u      [B,L,D]  当前输入信号
delta  [B,L,D]  每个时间、每个通道的步长
A      [D,N]    状态衰减参数
B      [B,L,N]  当前输入如何写入状态
C      [B,L,N]  当前状态如何读成输出
D_skip [D]      输入直通系数
output [B,L,D]
```

### 主要公式

```text
decay[n]  = exp(delta[t,d] * A[d,n])
update[n] = delta[t,d] * B[t,n] * u[t,d]
state[n]  = decay[n] * old_state[n] + update[n]
y[t,d]    = sum_n(state[n] * C[t,n]) + D_skip[d] * u[t,d]
```

白话解释：旧记忆先按 `decay` 衰减，当前输入按 `update` 写入，再用 `C` 从状态中读出结果；`D_skip` 允许输入直接绕过状态系统。

### 实现逻辑

`(B,D)` 的不同 group 可以并行，同一 group 内按 `L` 顺序推进。`A[d,:]` 搬入 UB 后复用，`state[N]` 保持在 UB，`B/C` 每步流入，最终只写出 `output`。

### 证据

- 主 scan shape：`B=1,L=1024,D=1536,N=16`。
- 历史 scan：torch_npu `237.7 ms`，vectorized custom `27.38 ms`。
- Mamba block：`185.72 -> 20.59 ms`，`9.02x`。
- 真实 MTO validation loop：8 个窗口，`1699.62 -> 166.38 ms`，`10.22x`。
- TorchAir 正式 shape：`27.952 ms`；custom `20.077 ms`，custom 仍快 `1.39x`。

不能说“完整训练后的 Mamba 在所有数据集提升 10.22x”；当前证明的是指定 block/validation 路径的执行等价性和性能。

## 第 6 页：S6ScanFused

### 与 SelectiveScan1D 的区别

SelectiveScan1D 接收已经生成的 `delta/B/C`；S6ScanFused 从输入 `x_t` 开始，在一个 layer 内同时计算步长、门控、候选内容和状态递推。

### 公式

```text
delta_t = softplus(x_t * Wd + bd)
decay_t = exp(-delta_t * a)
gate_t  = sigmoid(x_t * Wb + bb)
proj_t  = x_t * Wx + bx
h_t     = decay_t * h_(t-1) + gate_t * proj_t
```

### 测试范围

- `B=32,L=336,IN=11,H=64`，3 层 encoder。
- 输入和权重是确定性随机张量。
- 历史 eager 对照：单层 `136.56x`，3 层 encoder `44.15x`，max diff `5.22e-08`。

### TorchAir 结论

原始 `softplus` 表达缺 converter，不能直接完整成图。将 softplus 改写为数学等价、数值稳定的基础算子后：

- TorchAir `4.372 ms`；
- custom `0.674 ms`；
- custom 相对图路径 `6.48x`；
- max diff `4.47e-08`。

人工改写不是 TorchAir 对原始模型的自动优化，因此 PPT 使用“改写后可成图”口径。custom 直接实现 S6 递推语义，不需要为了缺失 converter 把 softplus 拆成另一套 PyTorch 表达。

## 第 7 页：AutoCorrFusedAggregate

### 在 Autoformer 中的位置

```text
Q/K/V projection -> lag score -> Top-K lag -> Softmax
                 -> 循环位移 value -> 加权聚合
```

Autoformer 不直接对每个时间点做普通 self-attention，而是寻找序列中最相关的周期位移 `lag`。

### 公式

```text
score[lag] = sum_t q[t] * k[(t+lag) mod L]
lags       = TopK(score, top_k)
weights    = Softmax(score[lags])
y[t]       = sum_i weights[i] * v[(t+lags[i]) mod L]
```

`lag=24` 可以理解为“当前模式与向后错开 24 步的模式很相似”。Top-K 选择最可信的几个周期，再按权重聚合 value。

### 历史模型路径

- 真实 MTO CSV，8 个窗口；`L=192,D_MODEL=64,heads=4,top_k=3`。
- framework validation loop `144.36 ms`，custom `14.35 ms`，`10.06x`。
- 聚合组件历史 eager `18.4998 ms`，custom `0.6593 ms`，`28.06x`。

### 图模式与当前 custom

原始路径在 `aten.roll` 缺 converter。将 roll 人工改写为 slice + concat 后：

- TorchAir `4.568 ms`；
- custom ACLNN `0.618 ms`；
- custom 相对图路径 `7.39x`。

这里的 TorchAir 不是原生 `roll` 代码直接运行，而是人工改写后的强基线。custom 直接实现 lag 聚合语义，不需要维护 `slice + cat` 版本。

当前 kernel 的指数近似已改为 ln2 范围缩减和六阶多项式，并补齐正负指数缩放。最终重建复验：模型量级 `0.610 ms / 7.45e-09`，标准差为 1 的宽动态输入 `0.633 ms / 1.43e-05`。正式 `7.39x` 使用同一轮 TorchAir/custom 的 `4.568/0.618 ms` 配对计时。

## 第 8 页：BatchSpdInvFp32 的性能价值

### 在 Koopa 中的位置

Koopa 把历史序列编码为 latent state，再通过 DMD 拟合线性状态转移矩阵：

```text
latent z -> snapshots x/y -> 求解 x*K=y -> 用 K 推进未来状态
```

原始实现：

```python
K = torch.linalg.lstsq(x, y).solution
```

`torch.linalg.lstsq` 在当前 NPU 软件栈不支持 NPU，只能回退 CPU。

### 数学变换

当 snapshot 数 `m` 小于 latent 维度 `E` 时，最小范数解可以写成：

```text
K = x^T * (x*x^T)^-1 * y
```

其中 `G=x*x^T` 只有 `m x m`，测试中的 `m` 为 3 到 7。前后两个大矩阵乘继续使用原生 NPU `bmm`，自定义算子只补齐小型 SPD Gram 矩阵求逆。

### 正式性能口径

| 场景 | 原路径 | custom NPU | 提升 | 精度 |
|---|---:|---:|---:|---:|
| DMD `B32,m7,E128` | CPU fallback `4341.75 ms` | `0.339 ms` | `12823.56x` | max diff `6.80e-08` |
| Koopa E2E `seq336,pred96,B32` | CPU fallback `5300.64 ms` | `8.80 ms` | `602.19x` | max diff `2.38e-07` |

这里的巨大倍率不是“NPU kernel 比另一个成熟 NPU 求解器快一万倍”，而是原始框架根本没有 NPU 实现，发生了设备到 CPU 的回退。

## 第 9 页：Koopa enablement 的实现逻辑

### 正式结论

- `torch.linalg.lstsq` 不支持 NPU，只能 CPU fallback。
- TorchAir 无法把一个底层不支持的 `linalg_lstsq` 路径编译成 NPU 完整图。
- `BatchSpdInvFp32` 提供 NPU 上的小型 SPD 求逆，使 DMD 主路径可以留在设备侧。
- Koopa 不参与 TorchAir/custom 倍率排名，只按 CPU fallback/custom NPU 对比。

### LDLT 分解

```text
G = L * D * L^T
G^-1 = L^-T * D^-1 * L^-1
```

LDLT 不需要开平方，适合 3x3 到 7x7 的小矩阵。kernel 将一整个 Gram 矩阵 DMA 到 UB，在 UB 内完成分解、三角逆和对称重组，再一次写回 GM。

### 为什么不直接在 GM 上逐元素更新

小矩阵的中间结果需要被后续步骤反复读取。放在 UB 可以降低访问成本，也能避免多核 scalar GM 读写带来的缓存可见性和同步问题。每个 batch 的 Gram 矩阵由一个独立工作单元处理。

### 需要避免的表述

不能说“Koopa 原始路径已经在 NPU 上运行，然后 custom 又比 TorchAir 快 602.19x”。正确说法是“原始 lstsq 不支持 NPU，只能回退 CPU；custom 补齐 NPU 能力，Koopa 指定端到端路径相对 CPU fallback 提升 602.19x”。

## 第 10 页：TirexSlstmCell

### 在 TiRex 中的位置

```text
hidden -> norm -> i/f/z/o 输入投影 -> sLSTM recurrent cell
       -> group norm -> residual -> FFN
```

本算子覆盖 sLSTM recurrent cell，不是完整 TiRex 全模型。

### 四个状态

- `h`：对外输出隐藏状态。
- `c`：累积内容。
- `n`：归一化累加器。
- `m`：指数门控的数值稳定尺度。

### 主要公式

```text
logfplusm = m_prev + logsigmoid(raw_f)
m_new = max(raw_i, logfplusm)
i = exp(raw_i - m_new)
f = exp(logfplusm - m_new)
c_new = f*c_prev + i*z
n_new = f*n_prev + i
h_new = o*c_new/n_new
```

`m` 的作用类似 log-sum-exp 稳定技巧，避免长序列指数运算溢出。

### 证据

- `B=64,S=64,H=512,heads=4`，共有 `64*4=256` 个独立 group。
- 使用真实 MTO 上下文和真实 NX-AI/TiRex 预训练权重。
- 历史 cell `14.31x`，完整 sLSTM layer `10.46x`。
- 原始 `log_sigmoid_forward` 缺 converter；人工稳定等价改写后 TorchAir `6.318 ms`，custom `3.542 ms`，custom 快 `1.78x`。

这里的 `6.318 ms` 同样来自人工改写后的 TorchAir，不是原始 TiRex 表达直接成图。custom 直接实现 sLSTM cell 语义，不需要维护 logsigmoid 等价改写。

`10.46x` 是 sLSTM layer，不是完整 12-block TiRex forecast E2E。

## 第 11 页：CfcScanFused

### 模型作用

CfC 是连续时间 RNN。每个时间步根据输入、旧状态和时间间隔生成两个候选状态，再用时间门控插值。

简化公式：

```text
candidate1 = tanh(W1*x_t + U1*h_prev + b1)
candidate2 = tanh(W2*x_t + U2*h_prev + b2)
gate       = sigmoid(time_a * delta_t + time_b)
h_t        = gate*candidate1 + (1-gate)*candidate2
```

### 实现价值

同一 batch 内时间步不能乱序，但 batch 之间可以并行。custom 将投影、激活、门控和状态更新放进单个 kernel 的时间循环。

### 证据

- `B32,L336,IN11,H64`，随机张量，3 层 encoder。
- 历史 component `84.83x`，3 层 encoder `70.26x`。
- TorchAir `8.090 ms`，custom `1.265 ms`，custom 相对图路径 `6.40x`。
- TorchAir 捕获 7,062 个 FX 节点，首次编译和运行 `154.54 s`。

## 第 12 页：CornnScanFused

### 模型作用

coRNN 把隐藏状态理解成二阶振荡系统中的位置 `y` 和速度 `z`，使用 IMEX 方式推进：

```text
z_t = z_prev + dt * F(x_t, y_prev, z_prev)
y_t = y_prev + dt * z_t
```

第二式必须使用已经更新后的 `z_t`，所以时间顺序不能改变。

### 证据

- `B32,L336,IN11,H64`，随机张量，3 层 encoder。
- 历史 component `56.73x`，3 层 encoder `54.15x`。
- TorchAir `3.579 ms`，custom `1.718 ms`，custom 相对图路径 `2.08x`。
- TorchAir 捕获 3,703 个 FX 节点，首次编译和运行 `92.83 s`。

性能来自合成 encoder，不应写成完整 coRNN forecasting 模型 E2E。

## 第 13 页：SruScanFused

### 模型作用

SRU 用输入侧投影提前生成候选内容和门控，递归部分比 LSTM 更轻：

```text
c_t = f_t*c_(t-1) + (1-f_t)*x_tilde
h_t = r_t*activation(c_t) + (1-r_t)*x_t
```

`f_t` 控制旧 cell 保留多少，`r_t` 控制输出使用非线性 cell 还是 input highway。

### 为什么融合收益大

单步数学计算很轻，框架 launch 和中间 tensor 的占比就更高。把 336 个时间步放进一个 kernel 后，调度成本显著下降。

### 证据

- `B32,L336,IN11,H64`，3 层 encoder。
- 历史 component `163.48x`，3 层 encoder `71.46x`。
- TorchAir `5.272 ms`，custom `0.884 ms`，custom 相对图路径 `5.97x`。
- 7,069 个 FX 节点，首次编译和运行 `165.43 s`。

## 第 14 页：UnicornnScanFused

### 模型作用

UnICORNN 同样维护位置/速度风格状态，但 recurrent 参数为对角形式。不同 hidden channel 可以独立计算，时间维仍必须递推。

简化形式：

```text
z_t = z_prev - dt * damping * z_prev + dt * activation(input_t + w_diag*y_prev)
y_t = y_prev + dt * z_t
```

### 证据

- `B32,L336,IN11,H64`，3 层 encoder。
- 历史 component `160.68x`，3 层 encoder `100.43x`。
- TorchAir `3.544 ms`，custom `0.521 ms`，custom 相对图路径 `6.80x`。
- 4,715 个 FX 节点，首次编译和运行 `250.93 s`。

## 第 15 页：LtcScanFused

### 模型作用

LTC 是连续时间网络。每个时间步内部还要执行 `K` 次 ODE unfold，测试中 `K=6`，因此总循环近似为 `L*K`。

简化理解：

```text
state = (old_state + input_drive + leak_term) / total_conductance
```

每次 unfold 都依赖上一次结果，形成双层顺序循环。

### 证据

- `B32,L336,IN11,H64,K6`，随机张量，3 层 encoder。
- 历史单层 `379.21 -> 4.60 ms`，`82.44x`。
- 历史 3 层 encoder `1134.71 -> 14.76 ms`，`76.90x`。
- TorchAir `10.227 ms`，custom `4.909 ms`，custom 相对图路径 `2.08x`。
- 24,878 个 FX 节点，首次编译和运行 `616.61 s`。

这是 3 层 encoder 结果，不包含 embedding、forecast head 和完整数据集评测。

## 术语速查

### scan

scan 指沿序列逐步更新状态：`state_t = f(state_(t-1), input_t)`。因为当前状态依赖上一步，所以同一序列的时间维必须顺序执行；不同 batch、channel 或 head 通常仍可并行。

### kernel launch

CPU/框架向 NPU 提交一次 kernel 执行任务。小算子很多时，真正计算很短，但 launch、排队和中间 tensor 读写会占据显著时间。

### CPU fallback

某个 PyTorch 算子没有 NPU 实现，框架把该计算转到 CPU。它会引入设备同步和数据搬运，并破坏全设备执行链。

### TorchAir fullgraph

TorchAir 尝试把完整 PyTorch 子图转换为 GE 图。`fullgraph=True` 要求全部节点都能捕获和转换；任何缺少 converter 的算子都会导致完整成图失败。

### FX 节点

PyTorch Dynamo 捕获图中的操作节点。循环被展开时，一个简单递推可能产生数千或数万个节点，导致编译时间和图复杂度上升。

### custom ACLNN 时间

本文的 custom 时间包含 ACLNN 描述符、executor、kernel launch、kernel 执行和同步，是用户调用自定义算子时看到的设备侧调用成本，不只是裸 kernel 指令时间。
